
- Task 1 freezes service defaults in a plain Python `Settings` class instead of adding a config framework, because the plan only requires stable env names/defaults and the repo currently relies on simple Python execution.
- `empty_status()` reads from `Settings()` by default so later server and storage layers can share one canonical empty-state contract without duplicating config-derived values.
- Web of Science remains opt-in in self-hosted mode: `sources.web_of_science.enabled` follows `ENABLE_WOS`, and its empty-state `status` is `disabled` unless the toggle is explicitly enabled.
- Task 2 persists JSON with temp-file-plus-`os.replace()` semantics in the target directory so status updates are atomic and downstream readers never observe a partially written `status.json`.
- `normalize_status()` now merges loaded payloads onto the frozen empty-state schema, which keeps persisted status forward-compatible while still forcing `storage.state_dir` to match the active runtime directory.
- The initial server only routes `GET /status` and returns 404 for every other path so later artifact-serving work can extend one narrow request handler without mixing unfinished route behavior into the first runnable skeleton.
- `service/server.py` keeps `/data` as the configured default but falls back to a temp-backed state directory only when creating the configured state layout raises `PermissionError`, which preserves container semantics while keeping the first local entrypoint runnable for verification.
- Task 4 isolates worker responsibilities in `service/worker.py`: one helper builds the `/app/main.py` argv, one helper decides whether WOS should be passed through, and one subprocess wrapper enforces `shell=False`, caller-supplied cwd, and timeout control for future scheduler/server integration.
- Failure metadata is exposed through small source-result helpers that preserve `last_success_at` and mark later failures as `stale`, so downstream status updates can distinguish a first-time failure from a failed refresh after a prior success.
- Task 7 keeps scheduling in a dedicated `ServiceScheduler` wrapper instead of embedding APScheduler calls directly in `service/server.py`, so later worker wiring can supply refresh and subprocess-shutdown callbacks without changing the HTTP server contract again.
- The server shutdown path now stops the scheduler before closing the HTTP server, and it clears `schedule.next_run_at` on shutdown so later `/status` work can report that no further cron runs will be accepted after `SIGTERM`.
- Task 8 keeps backward compatibility by leaving `main.py` and the GitHub Actions contract untouched while making service-mode cleanup explicit: the subprocess wrapper strips `citation_updated.flag` and `summary.md` from service staging directories after each run, so those legacy files remain available to direct CI/batch invocations but never become part of the self-hosted runtime contract.
- Task 5 publishes releases by copying only the staged `dist/` tree into `STATE_DIR/releases/<run-id>` and then replacing `STATE_DIR/current` with a newly created symlink in one `os.replace()` step, so failed validation or copy work leaves the previous public pointer untouched.
- Task 6 keeps artifact serving on a strict allow-list in `service/server.py`: `/citation.json`, `/all.svg`, `/review.svg`, and single-segment `/{author_pub_id}.svg`, which prevents directory traversal and arbitrary file reads while preserving the documented compatibility URLs.
- Missing `citation.json` data is treated as a service empty state with the documented `503 {"error":"no_data","message":"No successful refresh yet"}` response, while missing SVG artifacts stay plain `404` responses so badge consumers keep their existing semantics.

- Chose a single root `Dockerfile` rather than compose or extra deployment tooling to keep the packaging layer aligned with the task scope.
- Kept `requirements.txt` unchanged because the container can install the existing runtime dependencies as-is.
- Documented the Docker path as additive to GitHub Actions/GitHub Pages and noted that WOS stays disabled by default unless explicitly enabled later.
