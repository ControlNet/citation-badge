
- Implemented the initial `service/` contract layer with stdlib-only env parsing in `service/config.py` and an `empty_status()` helper in `service/state.py`.
- The empty `/status` payload now freezes the downstream schema for `service`, `schedule`, `storage`, and per-source state, with Google Scholar enabled by default and Web of Science defaulting to `disabled` when `ENABLE_WOS=0`.
- Gotcha: package version must be declared before re-export imports in `service/__init__.py` so `service.state` can safely read `__version__` during import.
- Task 2 adds `service/storage.py` as the stdlib-only runtime state layer, with a stable layout contract for `status.json`, `releases/`, and the future `current` release pointer under the caller-provided state directory.
- The status loader now treats missing, malformed, or partial persisted data defensively by falling back to the canonical `empty_status()` schema instead of raising during service startup.
- Follow-up fix: `normalize_status()` must preserve dict-shaped baseline sections when persisted nested values have the wrong type (for example `{"storage": "oops"}`), otherwise downstream field access can still raise despite top-level fallback handling.
- Task 3 adds `service/server.py` as the first runnable stdlib HTTP entrypoint, and it reuses `ensure_state_layout()` plus `safe_load_status()` so `/status` always comes from the persisted runtime contract instead of ad hoc response assembly.
- Practical bootstrap gotcha: the Docker-default `/data` path may be unwritable during local `python -m service.server` runs, so the server startup path needs a narrow fallback to a temp state directory on `PermissionError` rather than failing before `/status` is reachable.
- Follow-up correction: the temp-directory bootstrap fallback in `service/server.py` was removed because it was scope creep; server startup now uses the configured `STATE_DIR` only and preserves the frozen runtime contract.
- Task 4 keeps the service worker as a subprocess wrapper around `/app/main.py`, because `main.py` writes `dist/` relative to its working directory and the self-hosted runtime must stay additive rather than importing scraper logic in-process.
- Service-mode argv intentionally omits `--gen_summary` even though CI includes it, and `--wos` is only emitted when both the service toggle is enabled and a non-empty WOS identifier is present.
- Task 7 adds `service/scheduler.py` as the cron orchestration layer: it writes `schedule.cron`, `schedule.timezone`, `schedule.refresh_on_startup`, `schedule.overlap_policy`, and run timestamps back into `status.json` so `/status` can stay file-backed instead of growing ad hoc server state.
- Graceful shutdown in `service/server.py` now routes `SIGINT` and `SIGTERM` through a background shutdown thread so the server can stop the scheduler and then break out of `serve_forever()` without deadlocking the main thread.
- Workspace gotcha: APScheduler is not installed in the current editor/runtime environment, so `service/scheduler.py` keeps a tiny import-safe fallback for local verification while still preferring the real APScheduler implementation whenever the dependency from `requirements.txt` is available.
- Task 8 makes the CI/service boundary explicit in `service/worker.py`: service mode forwards Scholar inputs without rewriting them, treats `dist/` as the only public worker output, and removes `citation_updated.flag` plus `summary.md` after subprocess runs so later service code cannot depend on CI-only side files.
- Task 5 adds `service/promote.py` as the narrow release-promotion layer: staged runs are only promotable when `staged/dist/citation.json` parses successfully and `staged/dist/all.svg` is present, which blocks incomplete worker output from replacing the last public release.
- Follow-up correction for Task 7: the temporary APScheduler fallback shim and missing-import suppression were removed as scope creep, so `service/scheduler.py` now depends directly on the real APScheduler package declared in `requirements.txt`.
- Local verification now assumes a pixi-managed default environment under `.pixi/envs/default`, while Docker runtime remains on the container's system Python without adding an in-container virtual environment.
- Task 6 reuses `service.promote.current_release_path()` in `service/server.py`, so compatibility routes always resolve the promoted `current` release before reading `dist/citation.json` or SVG artifacts.
- The required SVG header check uses `curl -I`, so the stdlib handler needs explicit `HEAD` support in addition to `GET` even though the public contract is route-based rather than method-rich.
- Task 9 packaging is intentionally thin: a root `Dockerfile` on `python:3.10-slim` can install the existing requirements, run from `/app`, and launch `python -m service.server` without introducing pixi or an in-container virtualenv.
- Self-hosted docs should call out the additive Docker path, `/data` volume state, and the default Scholar-first / WOS-disabled runtime posture so the README matches the service defaults.
- Pixi deprecation noise was removed by switching the manifest header from `[project]` to `[workspace]` in `pixi.toml`.
- Task 7 remediation wires `CitationServiceHTTPServer` to a real `ServiceRuntime` refresh callback that stages each worker run in a private temp directory, reuses `main.py` via subprocess, promotes only validated `dist/` output, and leaves the previous `current` release untouched when the worker fails or staged output is incomplete.
- Graceful shutdown needs the runtime to treat an in-flight worker termination as a shutdown outcome before inspecting the subprocess return code; otherwise a SIGTERM-driven `-15` exit gets misreported as a normal stale refresh instead of `service.status="stopping"`.
- Restart normalization now matters for file-backed `/status`: if the last persisted `service.status` was `running` or `stopping`, startup should rewrite it to `ready` when a promoted release exists and `idle` otherwise so a new process does not inherit the previous process's terminal lifecycle state.
